说说Java中HashMap的原理？

hashmap 是一个哈希表，设计一个哈希表需要解决问题：存到哪，如何存（哈希函数），哈希冲突，哈希表的扩容

  1. 存到哪？
    1. 维护了一个node类型的数组，这个数组通常我们叫他哈希桶，每个桶可以存储一个或多个键值对。 node类包含了键，值，键的哈希，指向下一个的指针
  2. 如何存？
    1. 当你put方法时： 首先会调用 hashcode（）计算哈希值，后续还会再对哈希值进行处理
    2. 根据哈希值（n-1）& hash计算得出键值对在桶中的位置
    3. 根据计算出的位置使用equals判断是否存在哈希冲突
      1. 存在哈希冲突： 就使用链地址法解决（下文会提到）
      2. 不存在哈希冲突：
        1. 键重复： 覆盖掉原来的值
        2. 不重复： 插入新的键值对
    4. 判断是否需要扩容
  3. 如何解决哈希冲突
    1. 使用的是链地址法
    2. java7及之前，只使用链表，当发生哈希冲突时，以链表的形式将新的值追加到这个桶里。
      1. 如果一个桶的链表的长度过长，那么查询的时间复杂度就会从o（1）退化到o（n）
    3. java8新增了红黑树来优化查询效率：当一个桶中的链表长度达到一定的长度（默认为8），并且hashmap的容量达到64后就会将其转化为红黑树，这样时间复杂度就由o（n）转为o（logn），当节点数量小于6时，又会转为链表来节省内存
  4. 如何读取
    1. 与put类似，get方法： 先计算键的哈希值，并通过同样的方法计算出其在桶中的位置
    2. 查找键值对：
      1. 判断第一个是不是要找的
      2. 顺着链表/红黑树进行遍历，直到找到要找的，返回
  5. 扩容机制
    1. 哈希表的性能与： 初始容量（默认16）和加载因子（默认0.75）有关
    2. 当hashmap中的元素数量超过初始容量 *加载因子 ，就会进行扩容，重新计算元素在新哈希表中的位置，并进行移动

Java中的hashCode和equals方法之间有什么关系？

这两个方法在普通对象与集合类中的作用是不同的

在java官方文档里规定了这两个的关系：

  • 如果equals 方法为true ，则两个对象的hashcode必须为真，反之：如果两个对象的hashcode 相同，则equals不一定为真。

在普通对象中

  • hashcode 是根据内存地址获取该对象的哈希值，那其实这并没有什么用，因为普通对象也不需要依靠哈希值存储
  • equals 方法的默认行为是通过== 判断两个对象的引用地址是否相同。
    • 但是我们在对象的比较时，我们更希望：如果两个对象他们的属性，方法都相同，就认为这两个是同一个对象。因此我们使用equals方法时会将其重写，让他按我们的逻辑进行比较。
    • ==但是== 因为我们重写了equals方法，导致两个内存地址不同的对象的equals方法的比较也为true，而这两个对象的hashcode却不同。违反了官方文档的要求
      • 如果我们只是使用equals来进行比较对象是否相同，而不将其作为集合类的键，不重写hashcode是可以的。
        • 但是某些框架它依赖两个方法的一致性，因此重写最好
        • 后续如果要扩展到集合类，那也是不方便的
      • 如果我们要将其作为集合类的键，则必须重写，保证满足官方文档的要求，否则会导致hashmap等集合类的数据冲突混乱。

在集合类中

  • hashcode方法是获取键的哈希值，并根据哈希值来判断键值对在哈希表中桶的位置
  • equals： 在发生哈希冲突时，用于判断两个对象是否真的相等

比如在hashmap中

  • put方法： 首先使用hashcode方法，获取键的哈希值，并进行扰动
    • 然后根据这个值得到其在哈希表中桶的位置
    • 如果该位置已有元素，使用equals判断是否当前值与输入值相同

Java 中 ConcurrentHashMap 1.7和 1.8 之间有哪些区别?

Concurrenthashmap 在hashmap 的基础上增加了并发的安全性

存储与插入的区别：
java7及之前是分段锁， segment +HashEntry

  1. 采用segment这样一个结构来存储，每个segment可以视作一个独立的 哈希表，每个segment是独立的，可以并发访问不同的segment
  2. 在segment内部维护了hashentry 的数组，用于存储键值对。
  3. 那么有n个segment ，那么支持的并发量就为n
  4. 默认有16个segment，也就是默认支持16个线程同时访问

put插入流程：与hashmap是一样的，找segment的步骤

  1. 计算哈希值
  2. 定位到具体的segment， 对该segment进行加锁
  3. 在segment 内部：
    1. 定位到hashentry数组中的某个桶
      1. 如果桶为空，则直接插入
      2. 如果不为空，遍历查找key
        1. 有一样的，更新value
        2. 没有一样的，在链表尾部插入新元素
  4. 进行扩容检查

get流程：

  1. 计算hash值
  2. 找到对应的segment
  3. 遍历链表查找匹配的key
    1. 不需要加锁，因为hashentry的字段是volatile的，能保证可见性

java8 采用

  1. node 数组+链表+红黑树的结构，
  2. 取消了segment，改用： synchronized+CAS+volatile实现线程安全。
  3. 更细粒度的锁的控制，每个桶单独加锁

put的插入流程：

  1. 计算hash值
  2. 判断桶是否为空
    1. 如果为空，则使用CAS进行插入，如果成功则返回，否则进入下一步
    2. 桶不为空，则使用synchronized对桶的头结点进行加锁，然后进行插入

get流程：

  1. 与hashmap是一致的，无需加锁，node数组的val与next字段是volatile的

扩容的区别：
java7，采用的是segment，扩容时，每个segment独立扩容，不会影响其他

java8， 整个concurrenthashmap变为原来的两倍，使用cas确保线程安全

同时引入了渐进式扩容，多个线程将旧数据迁移到新数组中

什么是Java 的 CAS（Compare-And-Swap） 操作?

是硬件级别的原子操作，比较内存中的值是否为预期值，如果是，则更新为新值，否则不做修改

java只要是通过unsafe类提供的本地方法实现的，但是为了安全起见，java不允许直接使用unsafe类，而是使用原子类来间接使用CAS

优点：

  1. 无锁
  2. 适合轻量级同步场景
  3. 高效

缺点：

  • 会导致ABA问题：
    1. ABA问题： 线程看到某个值是A，但是中间被其他线程改为B，然后又改回了A
    2. 对CAS来讲始终都是A，它无法察觉到中间的变化，可能引起问题
    3. 解决：
      1. 使用带版本号的原子引用类：AtomicStampedReference
  • 自旋开销大
  • 只能保证单个变量的原子性

为什么JDK1.8对HashMap进行了红黑树的改动？

因为1.8之前，使用链表来解决哈希冲突，当链表足够长，那么查询的效率就从o（1）变为o（n）
因此1.8引入了红黑树，当链表长度大于等于8且数组长度大于64（数组长度小于64则扩容）时，将链表转为红黑树，这样查询效率就从o（n）变为了o（logn）

那为啥不直接使用红黑树，还要等链表过长才转换。
因为红黑树的节点是普通节点大小的两倍

JDK1.8对HashMap除了红黑树还进行了哪些改动?

  1. 扩容机制
    1. 改进了扩容时元素的迁移机制，在扩容过程中不再对每个元素重写计算，而是根据数组长度的高位来判断元素是否留在原位置，还是迁移到新的位置
  2. 哈希函数
    1. 1.7进行了四次异或， 1.8将key的哈希值的高16位与低16位进行了异或，使得哈希值的分布更均匀
  3. 头插法改为尾插法
    1. 多线程下，特别是扩容时，头插法会导致指针成环的问题
    2. 1.8改为了尾插法
