TCP和UDP区别

最核心的区别是： tcp 提供的是有连接的的服务， udp 是无连接的服务

tcp 是面向字节流的可靠服务 ，有拥塞控制 ， 只能 一对一 发送 所以多用于网页浏览和文件发送等场景
udp 是面向 数据报 的不可靠服务， 无拥塞控制， 他可以实现广播 。 所以多用于直播，游戏 等丢包影响不大场景

tcp 的三次握手 ， 为啥不能少一次

首先 客户端向 服务端发送 请求携带 SYN， 表示请求连接 。
服务端 发送SYN+ ACK 表示接收请求 。
客户端再发送ACK 表示明确 了。 连接建立成功。

为什么需要三次握手。 如果少一次握手。
如果少一次， 那么服务端发送SYN +ACK 则连接建立。
如果说一次握手中，客户端发送了SYN，但是这个请求由于网络原因问题，延迟很大。 客户端认为请求丢失了， 于是超时重传， 重发的SYN 到达并建立连接完成数据传输。 此时第一次发送的SYN请求到达了服务端， 于是请求再次建立， 但是客户端认为数据发送完毕。 于是这条连接的资源就被浪费了 。
如果是三次握手， 那么 第一次的SYN 到达服务端， 服务端发送SYN+ACK 后，客户端立即发送RST包，服务端收到后，立即释放资源

同步双方初始序列号（ISN），防止序列号欺骗（ISN 随机化防攻击）。

第一次： 客户端发送 自己的 ISN （seq =100）
第二次： 服务端 确认对方的ISN（ack =101） 并发送自己的ISN（seq = 300）
第三次： 客户端确认对方的ISN（ack = 301）

tcp 四次挥手， msl是什么， 为啥要等2msl ，为啥是四次

首先, 客户端向服务端 发送FIN 标识的请求， 服务接收后，发送ACK ，表示知道客户端无数据发送了，要关闭连接。 服务端发送完数据后， 也发送一个FIN 。 客户端接收后，发送ACK ，并等待2msl 的时间关闭连接，服务端接到ACK 后关闭连接

msl 是最大报文段的存活时间， 为啥要等2msl
第四挥手： 客户端发送ACK 。 如果ACK 在网络中丢失。 那么服务端超时重传FIN， 而此时客户端如果关闭了， 就无法给出ACK 回应， 那服务端就会一直处于连接状态。
为什么是2msl ，

  1. 因为从服务端触发超时重传再次发送FIN 开始到 客户端重新发送ACK 结束， 最大的时间是2msl 。
  2. 清除旧连接的报文
    1. 等待2msl 可以保证当前连接的所有报文从网络中消失 。 否则旧的报文会被新的连接接收

四次的原因：

  1. FIN（客户端）：关闭客户端→服务端的数据流
  2. ACK（服务端）：确认客户端方向关闭
  3. FIN（服务端）：关闭服务端→客户端的数据流（服务端可能还有数据要发）
  4. ACK（客户端）：确认服务端方向关闭

TCP粘包 与拆包

TCP本身不存在”粘包”，这是应用层视角的问题。，由于tcp 是面向字节流的连接，不维护消息的边界， 我们在应用层看到 我们在期望接收到完整业务时，发现多个信息粘在一起，为粘包或者一个消息被拆开 为拆包。

为什么会出现粘包：

  1. tcp 传输的数据是流式， 不保证一次send 对应一次recv
  2. 操作系统缓冲区机制
    发送端： Nagle 算法 默认会合并小的数据包 缓冲区机制
    接收端： 滑动窗口大， 可能一次接收多个包
    缓冲层堆积

解决：

  1. 固定消息的长度： 浪费空间，灵活性差
  2. 添加一个表示结束的标识 ： 需要考虑转义
  3. 消息头中添加一个标识长度的字段
  4. 使用自定义的协议：如gRPC

（x）tcp可靠性

可靠性是指 ： 1. 不会丢失 2. 按序 3. 不错乱（比如a连接的数据发给b连接了）

实现机制：

  1. 序列号 与确认应答
  2. 超时重传
  3. 滑动窗口
  4. 拥塞控制
  5. 错误检测与数据排序

（-）TCP nagle算法？

主要为了解决早期互联网中小数据包很多的问题。

核心思路：

  1. 如果之前的数据都发送了并且全部确认， 则发送新数据
  2. 如果还有没确认的数据，缓存新数据，等满足下面任意一项： 发送数据
    1. 累计数据达到最大报文长度
    2. 收到之前的ACK
    3. 应用程序禁用

你们项目是否是微服务架构？用的 RPC 还是 HTTP？

RPC 代表框架： grpc ， dubbo

优点：

  • 高性能 ：通常基于二进制协议 。
  • 开发高效： 接口定义语言（Protobuf） 可自动生成客户端与服务端代码，保持接口一致性
  • 高级特性： 内置流式通信，超时，熔断，负载均衡

缺点：

  • 技术耦合性强
  • 可观测性差： 二进制协议对开发者不透明

Http/rest 代表：spring cloud Feign RestTemplate

优点:

  • 通用性与互操作性：Http 互联网通用
  • 可读性好：json 对人类友好

缺点：

  • 性能开销较大：json 体积更大，序列化与反序列化开销比二进制更多
  • 协议开销大： 请求含有大量的头信息

内部服务使用 RPC

对外暴露使用HTTP

  • 如果是系统内部服务之间对性能、吞吐量要求很高的调用，我会优先选择 RPC，比如 gRPC。因为它性能极高，并且能利用框架内置的治理能力简化开发。

  • 如果是需要对外部（浏览器、移动端、合作伙伴）暴露的接口，我会选择通用的 HTTP/REST，因为它通用、防火墙友好且易于调试。

  • 在实际项目中，我们经常采用一种混合架构：微服务内部通过 gRPC 进行高效通信，同时通过一个 API 网关 将内部的 RPC 接口聚合、转换为 RESTful API 统一对外暴露。这样既能享受内部的性能优势，又能保证外部的通用性。

为什么需要 HTTP 协议？HTTP 和 TCP 的关系。

tcp 解决的是可靠传输的问题，它传输的内容本身是没有任何语义的二进制流。 而 http 协议在tcp 之上，定义了这些二进制流应该是什么格式， 代表什么含义，从而实现了应用程序之间的通信

  • TCP 好比修建了一条可靠的高速公路（保证数据包不丢、不乱序）。

  • HTTP 好比在这条公路上制定的交通规则（比如红灯停、绿灯行）和汽车的标准设计规范（比如方向盘、油门、刹车在哪）。没有规则和规范，即使路修得再好，车辆也无法高效、有序地通行。

为什么需要http ：
如果没有http ， 那么服务器收到一串客户端通过tcp 发送的二进制数据， 就读不懂含义，缺少源信息（如 浏览环境， 语言 等请求头） 效率低下（每次传输数据，都要tcp 三次连接，非常慢）

TCP 是基础，HTTP 是构建在其上的应用

  • TCP是基础，它提供了可靠的、端到端的字节流传输通道，解决了数据在传输过程中可能出现的丢包、乱序、错误等问题。但它传输的内容本身是没有任何业务含义的。

  • HTTP是构建在TCP之上的应用层协议。它赋予了TCP字节流具体的语义和规则，定义了客户端和服务器之间通信的格式（比如请求行、头部、正文）和含义（比如GET表示获取资源，POST表示提交数据，200表示成功）。

HTTP 长连接与短连接的区别和好处。

http 长连接 是一次tcp 连接上 ，可以进行多次请求-响应， 用完再关闭

http 短连接 是一次请求-响应后立即关闭tcp 连接

长连接的好处：

  1. 极大的降低了延迟，提升性能
    1. 一次tcp 连接的建立需要一次RTT（往返时间） 进行三次握手。 如果使用短连接， 那么现代的网页，包含多个资源js 图片等， 就会触发几十个RTT，特别慢
    2. 而使用长连接， 就省去了大部分的握手延迟
  2. 减少网络拥塞
    1. 使用更少的 SYN ACK 等包

但是长连接 也有缺点：

  1. 资源占用更多，虽然连接少了， 但是保持连接持续打开， 会占用服务器资源。如果大量空闲连接不关闭， 会耗尽服务器资源
  2. 解决：
    1. 超时机制：设置一个超时时间
    2. 最大请求限制： 处理完一定请求后关闭

演变：

  • HTTP/1.1 + 长连接：解决了连接数爆炸的问题，但浏览器为了进一步提升并行加载能力，通常会与同一个域名建立 6~8个 长连接（每个连接一次只能处理一个请求-响应），然后在这些连接上分发资源。这仍然不是最优解。

  • HTTP/2：将并行能力推向了极致。它只需要一个长连接，通过多路复用（Multiplexing） 技术，就可以在这个连接上同时处理上百个交错的请求和响应，效率得到了质的飞跃。但这一切的前提，仍然是这个连接是“长”的。

“HTTP长连接和短连接最根本的区别在于TCP连接的复用程度。

HTTP/1.0默认使用短连接，即每次请求-响应都会经历一次完整的TCP三次握手和四次挥手。这在请求大量资源的现代网页中会带来巨大的性能和资源开销。

而HTTP/1.1默认使用长连接，允许客户端和服务器在一次TCP连接上发送和接收多个HTTP请求/响应。这带来了三个核心好处：

  1. 大幅降低延迟：省去了重复的TCP握手/挥手时间。

  2. 减轻网络和服务器负担：减少了控制包和需要维护的连接状态。

  3. 为性能优化奠基：它是HTTP管道化和HTTP/2多路复用等高级技术的基础。

当然，长连接需要配合超时机制等来管理，防止空闲连接过度消耗服务器资源。

进程与线程

进程：

  • 进程是操作系统资源分配的基本单位 是一个运行中的程序
  • ，有自己独立的地址空间以及资源。
  • 进程间的通信方式： 管道，消息队列，共享内存 。

  • 统一进程内的线程共享进程资源，
  • 线程之间的通信就是直接读取共享的内存的数据即可。 但必须使用同步原语（互斥锁、条件变量等）保证数据一致性
  • 线程的创建开销小

用户态和内核态

为啥分用户态与内核态呢：

  1. 安全隔离
    1. 如果不隔离，一个恶意程序就可以执行特殊操作（硬盘格式化，关闭防火墙等）
  2. 资源统一管理： 如果两个进程同时写磁盘，就会造成数据混乱，而如果所有的硬件访问都需要内核进行代理， 就可以防止
  3. 系统稳定性
    1. 用户态崩溃： 只影响单个进程
    2. 内核态崩溃：整个系统宕机

是同一个进程的两种状态 ， 用来区应用程序与操作系统内核 的操作权限

主要区别在于权限级别：

  1. 用户态：应用程序在用户态下运行， 权限较低，不能直接访问硬件或者进行特权操作。 进行系统调用只能与内核态进行通信， 由内核代为执行敏感操作。 安全性高
  2. 内核态： 内核态是系统内核运行的模式， 权限高，可以直接访问硬件资源与特权操作。

当程序中进行系统调用或硬件中断时， 操作系统会切换到内核态

在内核态与用户态进行切换时，会触发上下文切换， 包括当前寄存器，状态和栈等信息，并加载内核态的状态 及处理系统调用 。 上下文切换非常消耗资源， 优化性能时， 避免不必要的系统调用与态切换

死锁如何解决

死锁的触发条件：

  1. 互斥
  2. 占有等待
  3. 不可抢占
  4. 循环等待

解决 ： 破坏掉其中一个条件

  1. 资源排序法：将资源标号，线程按序号来请求锁
  2. 一次性分配： 线程在操作前一次性申请所有需要的资源，如果失败，则释放所有资源
  3. 采用超时机制 （破坏不可抢占） redis 的分布式锁的做法
  4. 按固定顺序加锁（破坏循环等待）
  5. 使用读写锁
  6. 无锁编程 cas 内存屏障

cpu 100% 如何确定

整个过程分为四个阶段：初步确认、识别高负载进程、深入分析原因、解决问题

  1. 初步识别
    1. cpu 100% 可能是瞬间的峰值， 使用top 命令 查看 cpu 的状况，如果1-2分钟，持续高负载，就深入排查
    2. 还有观察内存， 如果内存也是持续100% ，可能是内存泄露导致的频繁交换 间接引发高cpu
  2. 快速复现
    1. 尝试关闭所有非必要程序，看cpu 是否回落， 如果回落则说明可能出现在某个程序
  3. 识别高负载进程
    1. 找出占用cpu 最高的具体进程（PID） 和程序名称
    2. 输入top 命令， 按P （大写） cpu使用率排序
    3. 记住第一列的PID 与第四列的进程名
  4. 分析原因
    1. 如果是 单个应用占用了100% cpu
      1. 检查程序的行为： java 使用jstack pid 生成线程转储， 检查是否有死锁
      2. 分析占用最大的线程
    2. 系统级问题（与程序相关的驱动，服务）
      1. 通过top 找到pid ， 使用systemctl status pid 查看关联的服务
  5. 解决：
    1. 临时修复： 结束高cpu的程序 ， 重启系统
    2. 永久修复
      1. 禁用非必要的驱动
      2. 修复程序逻辑避免死锁
    3. 增加检测机制
