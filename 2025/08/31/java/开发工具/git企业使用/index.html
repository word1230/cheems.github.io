git企业使用

开发一个新需求的流程：

  1. 使用gitclone / idea 拉去代码

  2. 新建分支 注意分支名称命名

  3. 新功能： feat/add-doc

  4. 修bug： fix/update-doc

  5. 快乐编码 不断git commit

  6. 拉去最新main分支代码， merge 到feat分支

  7. 可能需要处理冲突

  8. 处理冲突后， push feat 到远程

  9. 在网页端发起 pr 进行code review

  10. 通过后合并到main

merge

在代码开发完成后， 我们需要先从远程获取最新的main， 然后将其merge到本地feat 分支。

如果在这个过程中出现了冲突，则解决冲突， 然后再push

若无冲突， 直接push。

merge完成后，在push的过程中，main更新了，而且有冲突。

我们到网页端发起pr时， 会出现无法合并，因为存在冲突。

此时我们需要重复一遍前面的操作： 切换到main ，拉去最新，merge 到feat ，再次push，刷新网页端，发现可以合并

解决冲突

当我们push时失败了

  1. pull 代码， 解决冲突

  2. 对代码逐行选择

  3. 此时再次push ， 会将之前失败的push 一同提交

merge 是将两条分支合并， 生成了一个新的提交

加入git fetch的流程

  1. 拉取main分支最新代码 (**git checkout main && git pull**)
  2. 创建特性分支 (**git checkout -b feat/xxx**)
  3. 编码 -> commit (循环)
  • 【关键优化点】 在编码过程中，每天开始工作前或实现一个较大功能点后，可以定期执行一下4、5两步：
  1. **git fetch origin**：默默获取远程所有分支的最新状态，不会影响你的工作目录。
  2. **git merge origin/main**：将远程主分支的最新状态直接合并到你的当前分支。_(或者使用_ **_git rebase origin/main_**，效果更整洁，但初学者可先熟练掌握merge)
  3. 编码完成，进行最后的同步：再次执行 **git fetch origin** 和 **git merge origin/main**（或 **git pull origin main**）。
  4. 解决冲突（如果有），测试。
  5. push feat分支。
  6. 网页端创建PR。

commit 的规范

  • **原子性提交 (Atomic Commits)**：

  • 是什么：每次提交只做一个完整的、小的逻辑变更。例如，修复一个BUG和重构一个函数应该分成两次提交。

  • 为什么：便于回滚。如果某个提交引入了问题，你可以轻松地只回滚这个有问题的更改，而不影响其他好的修改。也让Code Review更容易，评审者可以清晰地看到你的思路变化。

  • 提交信息的规范性：

  • 你已经注意到了命名，但这非常重要，值得再强调。好的提交信息能让人一眼看懂历史。

  • 格式：**<type>(<scope>): <subject>**。例如：**fix(auth): handle null token exception**。

  • Body：如果需要，在主题后空一行，写详细描述，说明为什么要这个修改，而不是做了什么（代码本身能看出做了什么）。

分支命名

  • 分支命名：你已经做得很好了。除了 **feat/**、**fix/**，还有：

  • **docs/**：仅文档更新

  • **style/**：代码风格变动（不影响逻辑）

  • **refactor/**：代码重构

  • **test/**：增加或修改测试

  • **chore/**：构建过程或辅助工具的变动

  • 长期分支：除了临时的特性分支，公司通常会有一些长期存在的分支。

  • **main**/**master**：生产环境代码。

  • **develop**：下一版本的集成分支。很多团队会从 **develop** 拉特性分支，完成后合并到 **develop**，测试稳定后，才将 **develop** 合并到 **main** 上线。（这就是著名的 GitFlow 工作流，虽然现在简化了，但你需要了解）。

  • **release/***：发布分支，用于测试和修复发布前的最后bug。

  • 环境分支：有些项目会为不同环境创建分支，如 **production**, **staging**。
