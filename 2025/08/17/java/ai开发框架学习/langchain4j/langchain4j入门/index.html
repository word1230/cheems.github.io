聊天与模型部分

一、 llm的接口

目前是只有两种， 而且第一种已经不维护了， 所以这里只介绍第二种

  • ChatLanguageModel ： 可以接收多个ChatMessage 作为输入并返回单个AIMessage ，而且ChatMessage支持==多模态==的信息

除了这个还有其他的模型 的接口类：

  • EmbeddingModel： 嵌入模型
  • ScoringModel：排序模型 ， 所谓的rank模型
  • ImageModel： 生成与编辑图片的模型

二、 chatmessage 的类型

  • UserMessage ：用户提示词
  • AIMessage： AI的响应
  • SystemMessage：系统提示词
  • ToolExecutionResultMessage：调用工具后的返回结果

在最简单的情况下，我们可以向 chat 方法提供单个 UserMessage 实例。 这类似于第一个版本的 chat 方法，它接受 String 作为输入。 主要区别在于它现在返回的不是 String，而是 ChatResponse。 除了 AiMessage 外，ChatResponse 还包含 ChatResponseMetadata。 ChatResponseMetadata 包含 TokenUsage，其中包含有关输入包含多少令牌的统计信息 （您提供给 generate 方法的所有 ChatMessages）， 输出（在 AiMessage 中）生成了多少令牌，以及总计（输入 + 输出）。 您需要这些信息来计算给定 LLM 调用的成本。 然后，ChatResponseMetadata 还包含 FinishReason， 这是一个枚举，包含生成停止的各种原因。 通常，如果 LLM 自己决定停止生成，它将是 FinishReason.STOP。
创建 UserMessage 有多种方法，取决于内容。 最简单的是 new UserMessage("Hi") 或 UserMessage.from("Hi")。

==为什么需要多个chatMessage==
因为一次如果只能传递一个消息， 那就无法实现ai的记忆。 记忆模块后面讲

聊天记忆部分

关于记忆与历史 两个方面

  • 历史是指与用户对话的完整的信息 ，在前端的ui上可以看到的 。这个langchain4j 并没有提供
  • 记忆是指：记忆保存一些信息，这些信息呈现给LLM，使其表现得好像”记住”了对话。 记忆与历史有很大不同。根据使用的记忆算法，它可以以各种方式修改历史： 淘汰一些消息，总结多条消息，总结单独的消息，从消息中删除不重要的细节， 向消息中注入额外信息（例如，用于RAG）或指令（例如，用于结构化输出）等等。

一、 chatMemory

前面 我们提到了chatMessage 的类型， 一次ai 调用可以输入多个message ， 是为了实现多轮对话的记忆，但是自己写特别麻烦。
这里langchain4j 封装了一个chatMemory 接口， 他是chatmessage的容器（List）
具体的chatmemory实现有两种：

  • MessageWindowChatMessage
  • TokenWindowChatMemory

二、 淘汰策略 及具体实现

既然chatmemory 是一个list ， 那就有容量限制，同时大模型的上下文也是有限的，并且上下文越多消耗的token就越多，花费时间就越长
所以 如果满了就要淘汰

淘汰就是 基于滑动窗口进行淘汰

具体的chatmemory实现有两种：

  • MessageWindowChatMessage： 比较简单， 就是简单的滑动窗口， 淘汰掉旧数据，保留最近的几条。（具体会更复杂， 可能包含令牌）
  • TokenWindowChatMemory： 也是滑动窗口，但专注于保留最近的n个令牌，根据需要淘汰旧消息。TokenWindowChatMemory需要一个Tokenizer来计算每个ChatMessage中的令牌数。

三、 消息的持久化

默认情况下，chatMemory 在内存中存储chatmessage

如果要 实现持久化到数据库/文件， 可自行实现

文档： https://docs.langchain4j.info/tutorials/chat-memory#%E8%AE%B0%E5%BF%86%E4%B8%8E%E5%8E%86%E5%8F%B2

主要是实现三个方法：

  • updateMessages： 根据memoryid ， 来区分不同的对话， 然后将对话内容存入数据库
  • getMessages： 根据memoryid 来从数据库中获取数据
  • deleteMessage ： 根据memoryid删除数据库中的数据

每当向ChatMemory添加新的ChatMessage时，都会调用updateMessages()方法。 这通常在与LLM的每次交互中发生两次： 一次是添加新的UserMessage时，另一次是添加新的AiMessage时。 updateMessages()方法预期会更新与给定内存ID关联的所有消息。 ChatMessage可以单独存储（例如，每条消息一条记录/行/对象） 或一起存储（例如，整个ChatMemory一条记录/行/对象）。

当ChatMemory的用户请求所有消息时，会调用getMessages()方法。 这通常在与LLM的每次交互中发生一次。 Object memoryId参数的值对应于创建ChatMemory时指定的id。 它可以用来区分多个用户和/或对话。 getMessages()方法预期会返回与给定内存ID关联的所有消息。

当调用ChatMemory.clear()时，会调用deleteMessages()方法。 如果您不使用此功能，可以将此方法留空。

四、 对于SystemMessage 的特殊处理

对于系统提示词：

  • 一旦添加 总是保留
  • 如果出现相同的系统提示词的添加， 则忽略
  • 如果不相同 ，则替换
  • 一次只能保存一个

五、对于工具消息的特殊处理

如果包含ToolExecutionRequest的AiMessage被淘汰， 随后的孤立ToolExecutionResultMessage也会自动被淘汰， 以避免与某些LLM提供商（如OpenAI）出现问题， 这些提供商禁止在请求中发送孤立的ToolExecutionResultMessage。

  • 当生成下一个部分响应时：调用 onPartialResponse(String partialResponse)。 部分响应可以由单个或多个标记组成。 例如，您可以在标记可用时立即将其发送到 UI。
  • 当 LLM 完成生成时：调用 onCompleteResponse(ChatResponse completeResponse)。 ChatResponse 对象包含完整的响应（AiMessage）以及 ChatResponseMetadata。
  • 当发生错误时：调用 onError(Throwable error)。
